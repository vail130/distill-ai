# ADR-0002: v1.0 scope and post-v1.0 roadmap

**Status:** Accepted
**Date:** 2026-05-17
**Context milestone:** M11–M17 (pre-v1.0); supersedes the v1.1 grab-bag in TODO.md

## Context

[ARCHITECTURE.md § v1 scope](../../ARCHITECTURE.md#v1-scope) names the
v1.0 formats (`pytest`, `jest`, `gotest`, `generic`) and the three
deferred buckets (v1.1 = more formats, v1.2 = MCP, v1.3 = code
distillation). The TODO.md v1.1 list, written early, became a
grab-bag: `k8s`, `json` logs, `npm`/`yarn`/`pnpm` install output,
`cargo` test/build, `rspec`, `mocha`. Nothing ties those items
together except "we thought of them."

Separately, exploring opportunities for distill-ai outside test
output surfaced three high-fit candidates: a Go static-analysis
Format (`golangci-lint` + `go vet`), a Rust JSON-envelope Format
(`rustc` / `cargo` / `clippy --message-format=json`), and a
Markdown outline Format. Each is real value — each could
plausibly land "next" — but they don't all belong in the same
release.

The decision is twofold:

1. **What does v1.0 ship?** Specifically: should any of the three
   new candidates be pulled into v1.0 alongside M11–M17?
2. **What is the v1.1 theme, and what gets deferred further?**

Both questions are time-sensitive because M11 (pytest) is the next
in-flight milestone. Decisions made now shape the cut.

## Decision

### v1.0 ships exactly what is scoped today

M0–M10 are done; M11 (pytest), M12 (jest), M13 (envelope), M14
(config), M15 (library API), M16 (docs), M17 (release prep) land as
already scoped. **No new Formats join v1.0.** The
ARCHITECTURE.md v1 scope line — "`pytest`, `jest`, `gotest`,
`generic`" — is the v1.0 contract and does not change.

The README's pitch (`pytest 2>&1 | distill-ai`) and the project's
narrow value proposition ("distil test output for LLM consumption")
are protected. Coherence at v1.0 matters more than breadth.

### Post-v1.0 ordering

| Version | Theme                    | Contents                                                                                                                                            |
|---------|--------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|
| v1.1    | Static analysis & linting | M23 `golangci-lint-json` + `go vet`, M24 `cargo-json` (rustc / cargo build / cargo test / clippy via `--message-format=json`).                      |
| v1.2    | MCP server               | Already planned. Lets agents call distill-ai as a tool rather than a pipe; amplifies every Format shipped before it.                                |
| v1.3    | Code distillation        | Unchanged: M18 Go source, M19 multi-language (WASM tree-sitter), M20 agent-read wrapper, M21 AST diff. M22 (compiler errors) narrows — see below.   |
| v1.4    | Documentation formats    | M25 Markdown outline. Plus HTML readable-content extraction once it has had its own design pass (not scoped in this ADR).                           |
| v1.5    | More log / test formats  | The displaced original v1.1 list: `k8s`, structured-`json` logs, `npm`/`yarn`/`pnpm`, `rspec`, `mocha`. Re-prioritised after v1.0 usage signal.    |

### v1.1 is static analysis, not the previous grab-bag

The previous TODO.md v1.1 list ships in **v1.5**, not v1.1. The
items remain valuable but cluster differently: log/test formats are
v1.0's theme and a v1.1 that repeats it adds nothing the user can
tell apart from v1.0. Static analysis is a coherent new theme —
*structured findings about static artefacts*, not *runtime
failures*.

Both v1.1 Formats consume **upstream-structured JSON** (`golangci-lint
--out-format=json`, `cargo ... --message-format=json`). They reuse
`Event`, `Location`, `StackFrame`, the budget enforcer, dedupe,
collapse, and all three encoders without modification. Marginal
cost is the lowest of any new-Format work the project has shipped
so far.

### M22 narrows

`M22 — Compiler / build-error formats` currently lists `rustc`,
`cargo`, `tsc`, `go build`, `gcc`, `clang`. The Rust pieces move to
**M24 (v1.1)** because they share a JSON envelope with clippy and
shipping them apart makes no sense. The remaining `tsc` / `gcc` /
`clang` work stays in M22 under v1.3.

`go build` errors are handled today by `gotest` (M10 captures
`build_failure` Events). No separate format needed.

## Consequences

**Positive:**

- v1.0 ships sooner: no scope creep between M10 and M17.
- The v1.0 pitch stays narrow and testable. "Distils test output"
  is one sentence; "distils test output and also some lint output
  and some doc outlines" is three.
- v1.1 is thematically coherent. Reviewers and users can describe
  it in one phrase ("static analysis Formats") and the upstream
  JSON envelopes mean implementation cost is bounded.
- The MCP server (v1.2) inherits the value of every v1.0 and v1.1
  Format for free. Doing MCP before adding many more Formats means
  each subsequent Format ships agent-callable without retrofit.
- Code distillation (v1.3) stays where the architecture demands:
  after the binary-size question in ADR-0001 has been answered.

**Negative:**

- The previous v1.1 list moves out by three releases. Users who
  wanted `k8s` or `json` log Formats in the first post-v1.0 cut
  will wait longer. Mitigation: the `generic` Format handles both
  shapes at floor-quality today; v1.5 brings specific Formats when
  there is signal that demand justifies the maintenance burden.
- Decisions are made before v1.0 has shipped, which means the
  v1.1+ scope is a hypothesis. Real usage signal may reorder
  v1.1 / v1.2 (e.g., bring MCP forward, or pivot v1.1 to whatever
  Formats users actually paste through). This ADR documents the
  intent; v1.1 scoping at the time will re-check the hypothesis.
- M22's existing scope shrinks. Anyone with an in-flight branch
  expecting Rust under M22 needs to retarget to M24.

## Alternatives considered

### Ship one or two of the new Formats in v1.0

What it gives: broader ecosystem coverage at v1.0; fewer point
releases needed to reach feature-parity with internal expectations.

Why rejected: dilutes the pitch. v1.0 is the version most likely
to be evaluated, screenshotted, and cited; the README needs to say
one thing, well. "Adds Go lint output and Rust cargo output too"
muddies that. M11–M17 are scoped and on the critical path; new
Formats add risk without changing the v1.0 selling point.

### Make v1.1 the previous grab-bag and defer static analysis to v1.4+

What it gives: respects the original roadmap; minimises change.

Why rejected: the grab-bag is not coherent and was never properly
scoped. Items in it (`rspec`, `mocha`, `npm`, `pnpm`) overlap with
each other, share no implementation, and lack a unifying theme.
Static analysis is a single coherent theme with shared
implementation (JSON envelope parsers), high agent-pipe frequency,
and no current `generic` workaround that produces structured
findings.

### Bring MCP forward to v1.1, push static analysis to v1.2

What it gives: agent-tool-callable surface arrives sooner, which
amplifies every existing v1.0 Format.

Why rejected: weakly. This is the option most likely to be
selected during v1.1 scoping if real usage signal shows that
agents care more about MCP than more Formats. It is rejected today
because Formats are the *unique* value of distill-ai (no other
tool does this) whereas MCP is plumbing (many tools speak it).
Shipping the unique thing first, then the plumbing, is the safer
default. Revisit at v1.1 planning.

### Skip Markdown entirely

What it gives: removes a low-leverage Format from the roadmap.

Why rejected: Markdown outline is honestly low-cost and the M20
agent-read wrapper (v1.3) is the natural consumer. Keeping it on
the roadmap as v1.4 costs nothing and gives M20 something to
consume.

### Add HTML readable-content extraction at v1.4

What it gives: addresses the single largest non-test, non-log
context-window sink (agents pasting in API responses, status
pages, dashboard error pages).

Why deferred: the parser is heavy (full HTML5), the "what to keep"
heuristic is fuzzy, and the design has not had its own pass. Worth
doing — not worth committing to in this ADR. Track it as a v1.4
candidate alongside Markdown but scope it separately when v1.4
opens.

## Re-evaluation criteria

This decision should be revisited if any of the following changes:

- v1.0 ships and real agent-feedback shows the v1.1 hypothesis
  (static analysis) is wrong — e.g., users overwhelmingly pipe
  `kubectl logs` rather than `golangci-lint`. v1.1 scope rotates.
- `golangci-lint` or `cargo`'s JSON output schema becomes unstable
  or is deprecated in favour of a different envelope. Pin a schema
  version at scoping time; rescope if the upstream pulls the rug.
- The MCP ecosystem matures faster than predicted (clients
  stabilise, protocol settles), making v1.2's leverage outweigh
  v1.1's Format additions. Swap.
- A v1.0 user explicitly contributes a v1.5 Format (`k8s`, `json`
  logs, `npm`, etc.) before v1.1 lands. Merge on its own merits;
  this ADR is about ordering work the project does itself, not
  blocking contributions.
